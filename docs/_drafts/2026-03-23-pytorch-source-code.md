## TorchDynamo

 Dynamo's compilation must happen in Python because the entire tracing pipeline is          
  fundamentally a Python-level operation:
                                                                                             
  1. Symbolic execution of Python bytecode — symbolic_convert.py walks the user's Python     
  bytecode instruction-by-instruction, simulating execution with symbolic VariableTracker    

    objects. This requires understanding Python semantics (closures, exceptions, control flow, 
    attribute access, __dunder__ methods, etc.) — things that would be enormously complex to
    reimplement in C++.
  2. FX graph construction — The output is a torch.fx.GraphModule, which is a Python
  nn.Module with a Python-level IR (torch.fx.Graph). FX is inherently a Python framework.    
  3. Backend invocation — The backend (e.g., Inductor) is a Python callable. Inductor itself
  (compile_fx) does graph lowering, fusion, and code generation all in Python. User-defined  

    backends are also Python callables.
  4. Guard generation — Guards are generated by inspecting Python objects (types, values,    
  tensor metadata) and producing Python code/closures that check validity.                   

  In contrast, the C side handles the runtime hot loop — the part that runs on every single  
  Python frame, potentially millions of times per second:
  - Intercepting the frame (CPython eval frame hook)                                         
  - Evaluating guards (C++ TensorCheck, RootGuardManager)                                    
  - Dispatching to cached compiled code                  
                                                                                             

  This is the classic split: compile time is cold (runs once per cache miss, in Python where 
  expressiveness matters), runtime is hot (runs every frame, in C/C++ where microseconds     
  matter). Moving compilation to C++ would gain nothing — compilation already takes          
  milliseconds to seconds — while massively increasing code complexity.

## `torch/compiler/`

`torch/compiler/` is the **public API surface** for `torch.compile` and the compiler subsystem. It's a thin facade that re-exports and documents user-facing functionality from the internal `torch._dynamo` module.

### Structure

- **`__init__.py`** (~735 lines) — The main API module
- **`config.py`** — Compiler configuration options
- **`_cache.py`** — Cache artifact serialization/deserialization

### What it provides

The APIs fall into a few categories:

**Core compilation:**
- `compile()` — Delegates to `torch.compile()`
- `reset()` — Clears all compilation caches
- `set_stance()` — Controls compiler behavior mode (e.g. `"force_eager"`, `"eager_on_recompile"`)

**Graph control:**
- `allow_in_graph()` — Tells Dynamo to treat a function as opaque (don't trace inside it)
- `substitute_in_graph()` — Registers a Python polyfill for a C function so Dynamo can inline it
- `disable()` — Decorator to disable compilation on a function
- `nested_compile_region()` — Marks repeated subgraphs (e.g. transformer layers) for compile-once-reuse-many

**Guard filtering (unsafe):**
- `keep_portable_guards_unsafe()`, `skip_guard_on_inbuilt_nn_modules_unsafe()`, `skip_guard_on_all_nn_modules_unsafe()`, `skip_all_guards_unsafe()`, etc. — Functions passed via `guard_filter_fn` to reduce guard overhead at the cost of safety guarantees

**Introspection:**
- `is_compiling()` — True during `torch.compile` or `torch.export` tracing
- `is_dynamo_compiling()` — True only during Dynamo tracing
- `is_exporting()` — True only during `torch.export`
- `list_backends()` — Lists available compiler backends

**Caching & serialization:**
- `save_cache_artifacts()` / `load_cache_artifacts()` — Serialize/deserialize compilation caches
- `load_compiled_function()` — Load an AOT-compiled function from disk

## `torch/_dynamo/` (Python) vs `torch/csrc/dynamo/` (C/C++)

These two directories implement different layers of the same system, split along the **Python ↔ C boundary** for performance reasons.

### `torch/csrc/dynamo/` — The C/C++ Runtime (Hot Path)

This is the **performance-critical frame interception and cache lookup layer**. It operates at CPython interpreter level:

**`eval_frame.c` + `eval_frame_cpp.cpp`** — The heart of Dynamo's runtime:
1. Calls `_PyInterpreterState_SetEvalFrameFunc()` to install `dynamo_custom_eval_frame_shim` as CPython's frame evaluator
2. For **every Python frame**, the shim checks the thread-local callback:
   - `None` → disabled, run default
   - `False` → run-only mode (use cache, never compile)
   - Callable → Dynamo is active
3. `dynamo__custom_eval_frame()` (eval_frame_cpp.cpp:117) is the main decision function:
   - Calls `lookup()` to evaluate guards on cached compiled code
   - **Cache hit** → run the compiled bytecode via `dynamo_eval_custom_code()`
   - **Cache miss** → calls back into Python (`dynamo_call_callback`) to trigger compilation
   - Stores the resulting `CacheEntry` (guard + compiled code) for future lookups

**`extra_state.h/cpp`** — Stores per-code-object state on CPython's code object scratch space:
- `ExtraState` = linked list of `CacheEntry`s + `FrameState` (for dynamic shapes detection) + `FrameExecStrategy`

**`cache_entry.h/cpp`** — Each cache entry holds:
- `guard_manager` — guards that validate inputs
- `code` — the compiled bytecode to run when guards pass

**`guards.h/cpp`** — C++ guard evaluation (`TensorCheck`, `LocalState`). Guards check tensor metadata (dtype, device, shape, strides, dispatch keys) at C++ speed to avoid Python overhead on every frame.

### `torch/_dynamo/` (Python) — The Compiler Frontend (Compile Time)

This is the **tracing, analysis, and code generation layer**. It only runs when compilation is triggered (cache miss):

**`eval_frame.py`** — Python-side orchestration:
- `optimize()` / `_optimize()` — entry point from `torch.compile`, sets up the callback
- `_TorchDynamoContext` — manages entering/exiting the compilation context, calls `set_eval_frame()` (the C function)
- `OptimizedModule` — wraps `nn.Module` to route its forward through Dynamo

**`convert_frame.py`** — The callback that C calls on cache miss:
- `ConvertFrame.__call__()` receives the Python frame from the C layer
- Calls `_compile()` which creates a `symbolic_convert` trace, producing an FX graph

**`symbolic_convert.py`** — Bytecode-level symbolic execution. Walks the Python bytecode instruction-by-instruction, building up the FX graph.

**`output_graph.py`** — Holds the `OutputGraph` which accumulates the FX `GraphModule` during tracing, then calls `call_user_compiler(gm, example_inputs)` to hand it to the backend (e.g., Inductor).

**`guards.py`** — Python-side guard generation (produces the guard functions that the C++ `guard_manager` evaluates).

### The Flow Between Them

```
User calls compiled function
        │
        ▼
[C] eval_frame.c: dynamo_custom_eval_frame_shim
        │
        ▼
[C] eval_frame_cpp.cpp: dynamo__custom_eval_frame
        │
        ├─ lookup() guards ──→ cache hit ──→ [C] dynamo_eval_custom_code (run compiled bytecode)
        │
        └─ cache miss ──→ dynamo_call_callback ──→ calls into Python
                                                        │
                                                        ▼
                                              [Py] convert_frame.py: ConvertFrame.__call__
                                                        │
                                                        ▼
                                              [Py] symbolic_convert.py: symbolic trace
                                                        │
                                                        ▼
                                              [Py] output_graph.py: build FX GraphModule
                                                        │
                                                        ▼
                                              [Py] call_user_compiler(gm, inputs) → backend (Inductor)
                                                        │
                                                        ▼
                                              Returns GuardedCode (compiled code + guards)
                                                        │
                                                        ▼
                                              [C] create_cache_entry → stored in ExtraState
                                                        │
                                                        ▼
                                              [C] dynamo_eval_custom_code (run the new code)
```

**In short**: C handles the per-frame hot loop (intercept → check guards → dispatch), Python handles the cold path (trace → build graph → compile). The boundary is designed so that after initial compilation, the common case (cache hit) stays entirely in C/C++ for minimal overhead.

## `dynamo_call_callback` explained

### What it does

This function is the C++ → Python bridge. It's called from `dynamo__custom_eval_frame` on a **cache miss** (line 357 of eval_frame_cpp.cpp) to ask the Python-side compiler to trace and compile the frame.

### Step by step

```cpp
static py::object dynamo_call_callback(
    py::handle callback,          // the Python CatchErrorsWrapper callable
    THP_EVAL_API_FRAME_OBJECT* _frame,  // raw CPython interpreter frame
    FrameLocalsMapping* locals,   // optimized locals mapping
    CacheEntry* cache_entry,      // head of the cache linked list (or null)
    FrameState* frame_state)      // dict tracking dynamic shapes across calls
```

1. **Wraps the raw C frame** into a Python-visible object:
   ```cpp
   THPPyInterpreterFrame* frame = THPPyInterpreterFrame_New(_frame);
   frame->locals = (PyObject*)framelocals_mapping_to_dict(locals);
   ```
   CPython's internal `_PyInterpreterFrame` isn't a proper Python object. `THPPyInterpreterFrame` is a PyTorch-defined wrapper that exposes `f_code`, `f_globals`, `f_builtins`, `f_locals`, etc. as Python attributes so the Python-side tracing code can inspect the frame.

2. **Wraps the cache entry** as a pybind11 object (or None if this is the first compilation for this code object).

3. **Calls into Python** via pybind11:
   ```cpp
   py::object result = callback(frame, cache_entry_obj, frame_state);
   ```

### What `callback` actually is

The chain is:

```
torch.compile(fn, backend="inductor")
  → optimize() in eval_frame.py
    → _optimize_catch_errors(convert_frame.convert_frame(backend, hooks), ...)
      → catch_errors_wrapper(compile_fn, hooks)
        → CatchErrorsWrapper(callback=ConvertFrame(...), hooks=hooks)
```

So `callback` is a **`CatchErrorsWrapper`** instance. Its `__call__` (convert_frame.py:2246) receives exactly the three args the C++ passes:

```python
def __call__(self, frame, cache_entry, frame_state) -> ConvertFrameReturn:
```

It does skip checks (skipfiles, dispatch modes, already-started frames), then calls into `ConvertFrame.__call__()` which runs `_compile()` → symbolic tracing → FX graph → `call_user_compiler(gm, example_inputs)` → backend (Inductor).

### What comes back

The return is a `ConvertFrameReturn` dataclass with three fields:

| Field                 | Purpose                                                      |
| --------------------- | ------------------------------------------------------------ |
| `frame_exec_strategy` | `FrameExecStrategy(cur_action, recursive_action)` — tells C what to do for this frame and recursive child frames |
| `apply_to_code`       | Whether to persist this strategy for all future frames with the same code object |
| `guarded_code`        | The compiled code + guards (or `None` if skipping)           |

Back in C++ (eval_frame_cpp.cpp:359-362), these are unpacked:
```cpp
new_strategy = callback_result.attr("frame_exec_strategy").cast<FrameExecStrategy>();
apply_to_code = callback_result.attr("apply_to_code").cast<bool>();
guarded_code = callback_result.attr("guarded_code").ptr();
```

If `guarded_code` is not None, C creates a new `CacheEntry` from it (guard_manager + compiled bytecode), stores it in `ExtraState`, and immediately runs the compiled code via `dynamo_eval_custom_code`. On subsequent calls, `lookup()` finds this cache entry and runs it directly without ever calling back into Python.



`ConvertFrame.__call__()` is called in exactly **one place**: `CatchErrorsWrapper.__call__()` at convert_frame.py:2334:

```python
result = self._torchdynamo_orig_backend(
    frame, cache_entry, self.hooks, frame_state, skip=1
)
```

Here `self._torchdynamo_orig_backend` is the `ConvertFrame` instance (set at line 2243).

### The full call chain on a cache miss:

1. **C++**: `dynamo__custom_eval_frame` → `dynamo_call_callback(callback, frame, locals, cache_entry, frame_state)` (eval_frame_cpp.cpp:357)
2. **Python**: `CatchErrorsWrapper.__call__(frame, cache_entry, frame_state)` (convert_frame.py:2246) — does skip checks (skipfiles, dispatch modes, already-started frames)
3. **Python**: `ConvertFrame.__call__(frame, cache_entry, hooks, frame_state)` (convert_frame.py:2059) — wraps with error handling/suppression, calls `self._inner_convert(...)` at line 2070
4. **Python**: `self._inner_convert` is a `ConvertFrameAssert` instance (created at line 2047 via `convert_frame_assert()`), which does the actual tracing → FX graph → backend compilation

So the layering is:
- **`CatchErrorsWrapper`** — skip checks, DDP handling, acquire compile lock
- **`ConvertFrame`** — error handling/suppression, counters
- **`ConvertFrameAssert`** (`_inner_convert`) — the real work: bytecode analysis, symbolic tracing, graph construction, calling the backend

### where does dynamo generate the FX graph and hand it to the TorchInductor

Here's the handoff flow from Dynamo to Inductor:

### The Critical Handoff

The single handoff point is in **`torch/_dynamo/output_graph.py`** in `_call_user_compiler()` (~line 2678):

```python
compiled_fn = compiler_fn(gm, example_inputs)
```

Where `gm` is the finalized FX `GraphModule` and `compiler_fn` is the Inductor backend.

### Call Chain

```
OutputGraph.compile_and_call_fx_graph()        # output_graph.py:2318
  ├─ gm = _make_graph_module(root, self.graph) # Finalize FX graph into GraphModule
  └─ OutputGraph._call_user_compiler(gm, ...)  # output_graph.py:2636
       ↓
     compiler_fn(gm, example_inputs)           # THE HANDOFF (line ~2678)
       ↓
     inductor backend (backends/inductor.py)    # Registered via @register_backend
       ↓
     compile_fx(gm, example_inputs)            # compile_fx.py:2497 — Inductor entry
       ↓
     compile_fx_inner()                        # compile_fx.py:787 — actual codegen
```

### Key Files

| Step                         | File                                                         |
| ---------------------------- | ------------------------------------------------------------ |
| Backend registration         | `torch/_dynamo/backends/inductor.py` — registers `compile_fx` as the "inductor" backend |
| Graph finalization + handoff | `torch/_dynamo/output_graph.py` — `compile_and_call_fx_graph()` and `_call_user_compiler()` |
| Inductor entry               | `torch/_inductor/compile_fx.py` — `compile_fx()` takes ownership of the graph, orchestrates AOT Autograd and codegen |
| Inner compilation            | `torch/_inductor/compile_fx.py` — `compile_fx_inner()` does the actual kernel/code generation |

The graph finalization happens in `compile_and_call_fx_graph()` where the traced operations are packaged into an `fx.GraphModule` via `_make_graph_module()`, metadata is attached (compile IDs, param sources, stack traces), and then the backend is invoked.

## `dynamo` Implementation

It hooks into CPython's frame evaluation to intercept Python execution, compile functions, cache compiled variants, and validate them with guards.

------

### Architecture Overview

```
Python function called
    ↓
dynamo__custom_eval_frame_shim()        [eval_frame.c]
    ↓
dynamo__custom_eval_frame()             [eval_frame_cpp.cpp]
    ↓
lookup() → check guards on cache       [extra_state.cpp]
    ├─ HIT  → dynamo_eval_custom_code() → run compiled code
    └─ MISS → dynamo_call_callback()    → invoke Python compiler
                  ↓
              create_cache_entry()      → store in ExtraState
                  ↓
              dynamo_eval_custom_code() → run new compiled code
```

### Key Data Structures

| Structure             | Purpose                                                      |
| --------------------- | ------------------------------------------------------------ |
| `ExtraState`          | Cache container attached to each code object                 |
| `CacheEntry`          | Single compiled variant with its guard manager and bytecode  |

### Key Implementation Details

1. **Version compatibility**: Extensive `#if IS_PYTHON_3_XX_PLUS` guards across all files for CPython 3.10–3.14+
2. **Guard hot path**: Guards are pre-compiled via `convert_to_root_guard_manager()` to avoid Python calls during cache lookup
3. **Lazy dict construction**: `FrameLocalsMapping` avoids `frame.f_locals` overhead — guards use direct indexed access
4. **LRU cache**: Cache entries are reordered on hit; configurable via `_set_lru_cache()`
5. **Shadow frames**: Compiled code runs in a new frame with copied args/locals/cells, preserving the original frame state
6. **Profiling**: All compilation and execution wrapped in `at::RecordFunction` for tracing/observability

## The PyTorch Dispatcher

### Call chain

```
torch.add(x, y)                                          [Python]
  │
  ▼
THPVariable_add(self_, args, kwargs)                      [generated: python_torch_functions_0.cpp]
  │  parse args, release GIL
  ▼
at::_ops::add_Tensor::call(self, other, alpha)            [generated: ATen/ops/add_ops.h]
  │  static op handle via Dispatcher::singleton().findSchemaOrThrow()
  ▼
Dispatcher::call<Return, Args...>(op, args...)            [aten/src/ATen/core/dispatch/Dispatcher.h:776]
  │
  ├─ 1. Extract dispatch keys from all tensor args:
  │     ks = x.key_set() | y.key_set() | TLS overrides
  │
  ├─ 2. Lookup kernel in dispatch table:
  │     kernel = op.lookup(ks)          // O(1) table index
  │
  └─ 3. Call the kernel:
        kernel.call<Return, Args...>(op, ks, args...)
          │
          ▼
        Actual CPU/CUDA/Autograd kernel
```

### Key components

**Dispatch keys** (`c10/core/DispatchKey.h`) — A bitset on every tensor identifying what it needs: `CPU`, `CUDA`, `AutogradCPU`, `Functionalize`, `Python`, etc. The union of all input tensors' keys, modified by thread-local overrides (grad mode, torch dispatch modes), determines which kernel runs.

**The Dispatcher** (`aten/src/ATen/core/dispatch/Dispatcher.h`) — A singleton that owns a table of all registered operators. The core `call()` method (line ~776) does three things:
1. `dispatchKeyExtractor.getDispatchKeySetUnboxed(args...)` — unions `key_set()` from all tensor args, applies TLS include/exclude
2. `op.lookup(dispatchKeySet)` — indexes into a precomputed dispatch table (`OperatorEntry.h:182`)
3. `kernel.call(...)` — invokes the selected kernel

**OperatorEntry** (`aten/src/ATen/core/dispatch/OperatorEntry.h`) — Per-operator dispatch table. Fixed-size array indexed by dispatch key. Populated at static init time via `registerImpl()`.

**DispatchKeyExtractor** (`aten/src/ATen/core/dispatch/DispatchKeyExtractor.h`) — Computes the effective dispatch key set:
```cpp
// Union all tensor keys, apply TLS overrides
ks = (tensor_keys | local.included_) - local.excluded_
```

**Redispatch** — Kernels like the autograd wrapper call `redispatch()` to forward to the next key. E.g., `AutogradCPU` kernel records the backward graph, then redispatches to the `CPU` kernel for the actual compute.

### Where kernels are registered

Kernel registration happens at static initialization (before `main()`). For native ops, generated code in `build/aten/src/ATen/RegisterCPU.cpp`, `RegisterCUDA.cpp`, etc. calls `Dispatcher::registerImpl()` to populate the dispatch table for each (op, dispatch key) pair.

torch.add(x, y)  →  Dispatcher  →  CPU/CUDA kernel

Inductor fuses this into **one** Triton kernel that does `relu(x + 1)` in a single GPU pass. The individual `torch.add` and `torch.relu` dispatcher calls **never happen** — they're replaced by a direct call to the generated Triton kernel.

**For CPU (C++/OpenMP):**
Similarly, Inductor generates fused C++ code compiled via `CppCodeCache`, again bypassing per-op dispatch.

## How Inductor interacts with the dispatcher

### Short answer: Inductor mostly **bypasses** the dispatcher.

### The two categories of ops in Inductor's output

**1. Fused ops → Direct Triton/C++ kernels (no dispatcher)**

Inductor's main job is **fusion and codegen**. When it sees pointwise/reduction ops in the FX graph, it:

1. **Lowers** them from ATen ops to Inductor's loop-level IR (`torch/_inductor/ir.py`)
2. **Fuses** compatible ops via the scheduler (`torch/_inductor/scheduler.py`)
3. **Generates code** — Triton kernels for CUDA, C++/OpenMP for CPU (`torch/_inductor/codegen/`)

The generated code calls Triton/C++ directly. For example, `add + relu` becomes a single Triton kernel:

```python
@triton.jit
def triton_fused_add_relu(in_ptr0, out_ptr0, xnumel, ...):
    x = tl.load(in_ptr0 + xindex)
    x = x + 1
    x = tl.maximum(x, 0)  # relu
    tl.store(out_ptr0 + xindex, x)
```

No `torch.add`, no `torch.relu`, no dispatcher — just raw memory loads, compute, stores.

**2. Non-fusible ops → `ExternKernel` (goes through the dispatcher)**

Some ops can't be lowered to Inductor's IR (e.g., `torch.mm`, `conv2d`, custom ops). These become `ExternKernel` nodes (`ir.py:5959`):

```python
class ExternKernel(InputsKernel):
    """Kernels not directly lowered to Inductor Loop Level IR,
    such as custom operators, or aten operators which we fallback to."""
```

For these, the generated wrapper code emits a call like:
```python
extern_kernels.mm(buf0, buf1, out=buf2)
```

Which ultimately calls `torch.ops.aten.mm` — and **that** goes through the dispatcher to reach cuBLAS/MKL.

### The generated wrapper

`GraphLowering.compile_to_module()` (`graph.py:2492`) produces a Python module that orchestrates everything:

```python
# Simplified generated wrapper code
def call(args):
    buf0 = args[0]
    buf1 = empty_strided(...)                    # allocate output
    triton_fused_add_relu[grid](buf0, buf1, ...) # direct Triton call (no dispatcher)
    buf2 = empty_strided(...)
    extern_kernels.mm(buf1, args[1], out=buf2)   # goes through dispatcher → cuBLAS
    return buf2
```

### Summary

```
                    Inductor compilation
                           │
              ┌────────────┴────────────┐
              │                         │
     Fusible pointwise/          Non-fusible ops
     reduction ops               (mm, conv, custom)
              │                         │
              ▼                         ▼
     Generated Triton/C++        ExternKernel nodes
     kernels (direct call)       in wrapper code
              │                         │
              ▼                         ▼
     Raw GPU/CPU compute         torch.ops.aten.*
     (NO dispatcher)             (THROUGH dispatcher → cuBLAS/MKL)
```

The performance win comes from the left path: fusing many small ops into one kernel eliminates both dispatcher overhead **and** kernel launch/memory traffic overhead.

## TorchDynamo: Device-agnostic

Dynamo has essentially **zero** hardware-specific code. It traces Python bytecode into an FX graph — the graph records `aten.add.Tensor`, not "CUDA add" or "CPU add". Device information is carried on the tensor objects themselves but Dynamo doesn't branch on it.

## TorchInductor: Heavily hardware-specific

Inductor has per-device code at multiple levels:

### 1. Codegen backends (the biggest split)

```
torch/_inductor/codegen/
├── triton.py                    # NVIDIA/AMD GPU — generates Triton kernels
├── cpp.py                       # CPU — generates C++/OpenMP code
├── mps.py                       # Apple Silicon — generates Metal shaders
├── cuda/                        # NVIDIA-specific
│   └── device_op_overrides.py   #   CUDA stream/device guard codegen
├── rocm/                        # AMD-specific
│   ├── ck_template.py           #   Composable Kernel (AMD's CUTLASS equivalent)
│   ├── ck_universal_gemm_template.py
│   └── rocm_template.py         #   ROCm-specific template kernels
├── cutlass/                     # NVIDIA CUTLASS GEMM templates
├── xpu/                         # Intel GPU
│   └── device_op_overrides.py   #   XPU stream/device guard codegen
└── mtia/                        # Meta Training & Inference Accelerator
```

### 2. Scheduling (how ops are fused and ordered)

| Scheduler                                                 | Device         | What it does                                                 |
| --------------------------------------------------------- | -------------- | ------------------------------------------------------------ |
| `TritonScheduling` (triton.py:6026)                       | NVIDIA/AMD GPU | Fuses ops into Triton kernels, manages shared memory, grid sizing |
| `CppScheduling` (cpp.py:4655)                             | CPU            | Fuses into vectorized C++ loops with OpenMP threading        |
| `CUDACombinedScheduling` (cuda_combined_scheduling.py:34) | CUDA           | Delegates between Triton and CUTLASS schedulers              |

### 3. `DeviceOpOverrides` (common.py:326)

Each device registers how to generate device-specific boilerplate:

```python
class CUDADeviceOpOverrides(DeviceOpOverrides):

    def synchronize(self):       return "torch.cuda.synchronize()"
    def device_guard(self, idx): return f"torch.cuda._DeviceGuard({idx})"

class CpuDeviceOpOverrides(DeviceOpOverrides):
    def set_device(self, idx):   return ""   # no-op for CPU
    def synchronize(self):       return ""

class XPUDeviceOpOverrides(DeviceOpOverrides):
    def set_device(self, idx):   return f"torch.xpu.set_device({idx})"
```

### 4. Wrapper codegen (the Python "glue" code)

```
cpp_wrapper_cpu.py     # CPU: generates C++ wrapper calling vectorized kernels
cpp_wrapper_gpu.py     # GPU: generates C++ wrapper calling Triton/CUTLASS
cpp_wrapper_mps.py     # MPS: generates wrapper for Metal kernels
```

### 5. GEMM template libraries

- **NVIDIA**: `codegen/cutlass/` — CUTLASS templates for matmul, generates CUDA C++
- **AMD**: `codegen/rocm/ck_template.py` — Composable Kernel templates
- **CPU**: `codegen/cpp_gemm_template.py`, `cpp_micro_gemm.py` — vectorized CPU GEMM with AVX/VNNI intrinsics

### Concrete example

For `torch.mm(A, B)` on different hardware, Inductor generates completely different code:

- **NVIDIA GPU**: May select a CUTLASS GEMM template or a Triton matmul kernel, with tile sizes tuned for the SM architecture
- **AMD GPU**: May select a Composable Kernel template (`ck_universal_gemm_template.py`)
- **CPU**: Generates C++ with micro-kernels using AVX512/AMX intrinsics (`cpp_micro_gemm.py`), or falls back to MKL via `extern_kernels.mm`
- **Apple MPS**: Falls through as an `ExternKernel` calling the Metal Performance Shaders backend

```
Dynamo:    Python bytecode → FX graph (device-agnostic)
                │
                ▼
Inductor:  FX graph → device-specific codegen
                │
    ┌───────────┼───────────┬──────────┐
    ▼           ▼           ▼          ▼
  Triton      C++/OMP    CUTLASS/CK   Metal
  (GPU)       (CPU)      (GEMM)       (MPS)
```

The clean separation means adding a new hardware backend mostly means adding a new scheduling + codegen module under `torch/_inductor/codegen/` — Dynamo doesn't need to change at all.

## When cuBLAS is used for CUDA matmul

The selection happens in `torch/_inductor/kernel/mm.py`. For a `torch.mm` call, Inductor builds a **list of candidate implementations** and then **autotunes** to pick the fastest:

### The candidates (mm.py:399-475)

```python
templates_to_use = []

# 1. ATen/cuBLAS — always included unless max_autotune is on
if use_aten_gemm_kernels():
    templates_to_use.append(aten_mm)          # → torch.mm → dispatcher → cuBLAS

# 2. Triton templates — hand-written Triton matmul kernels
if use_triton_template(layout, ...):
    templates_to_use.append(mm_template)
    templates_to_use.append(persistent_tma_mm_template)  # TMA variant
    templates_to_use.append(blackwell_ws_persistent_device_tma_mm_template)  # Blackwell

# 3. CUTLASS — NVIDIA's C++ GEMM library
if use_cutlass_template(layout, m, n, k):
    CUTLASS3xGemmTemplate.add_cutlass_gemm_choices(...)

# 4. CK — AMD's Composable Kernel
if use_ck_gemm_template(layout, m, n, k):
    CKGemmTemplate.add_ck_gemm_choices(...)

# 5. C++ GEMM — for CPU
if use_cpp_gemm_template(layout, mat1, mat2):
    CppGemmTemplate.add_choices(...)
```

### cuBLAS is used when:

**1. Default mode (no `max_autotune`)** — cuBLAS is the **only** choice.

`use_aten_gemm_kernels()` (utils.py:2516) returns `True` when `max_autotune` and `max_autotune_gemm` are both `False`. In this case, `aten_mm` (which calls `torch.mm` → dispatcher → cuBLAS) is the sole candidate — no Triton or CUTLASS templates are even considered.

This is the default for `torch.compile()`.

**2. With `max_autotune` enabled** — cuBLAS **competes** against other backends.

When you use `torch.compile(mode="max-autotune")`, all applicable candidates are benchmarked via `autotune_select_algorithm`. cuBLAS is still included as `aten_mm` and may win if:
- The matrix sizes happen to be well-optimized in cuBLAS
- Triton/CUTLASS templates don't have good configs for that shape

**3. cuBLAS always wins for certain patterns:**
- Very large GEMMs where cuBLAS has hand-tuned assembly kernels
- Ops like `torch.addmm` with specific alpha/beta values that map to cuBLAS's fused interface
- When Triton templates are excluded (e.g., Triton not installed, or shape heuristics skip them)

### In short

| `torch.compile` mode     | What runs matmul                                      |
| ------------------------ | ----------------------------------------------------- |
| Default (`mode=None`)    | **Always cuBLAS** via `aten_mm`                       |
| `mode="max-autotune"`    | **Best of cuBLAS vs Triton vs CUTLASS** (benchmarked) |
| `mode="reduce-overhead"` | **Always cuBLAS** (same as default, plus cudagraphs)  |

The `aten_mm = ExternKernelChoice(torch.mm, ...)` at mm.py:130 is the cuBLAS path — it emits `extern_kernels.mm(...)` in the generated wrapper, which calls `torch.mm`, goes through the dispatcher, and lands in cuBLAS.s
