## 一。机器学习框架的两大模式

引言首先将现代机器学习框架分为两类：

- **Eager Mode（动态/即时执行模式）：** 如 PyTorch 和 JAX。采用“定义即运行”的方式，代码在调用时立即执行。
- **Graph Mode（静态图模式）：** 如 TensorFlow、Caffe 和 Theano。采用“先定义后运行”的方式，需要先构建完整的计算图，然后再统一执行。



## 二。PyTorch1.0的缺点以及尝试过的方案

PyTorch1.0的Eager Mode相比较于采用图模式的Tensorflow等框架，缺点在于很难在图这一抽象层上做优化。在eager mode下，由于框架一次只能看到一个算子，无法自动进行跨算子边界的优化（如算子融合或调度）。



**早期方案的局限性**

   1. **灵活性与图表示的冲突**：

      与传统的“静态图框架”（如 TensorFlow、Caffe）不同，PyTorch 属于“动态模式（Eager Mode）”。用户可以在模型中嵌入任意的 Python 代码，包括非 PyTorch 库（如 NumPy、logging）、复杂的 Python 构造（类、闭包、异常）以及数据流控制。

   2. **转换瓶颈**：

      由于 PyTorch 允许频繁地将 Tensor 转换为 Python 原生类型（通过 `.item()` 或 `.tolist()`），这些行为很难被映射到一个固定的、不可变的图抽象（Graph Abstraction）中。

   3. **以往方案的权衡**：

      早期的图捕获尝试往往需要在“灵活性”和“性能”之间做选择。为了捕获图，这些方案通常会给用户体验施加诸多限制。虽然 **TorchDynamo** 最终解决了这种张力，但第 2 部分通过对比先前技术，为 PyTorch 2 的必要性提供了背景。

------

**代码示例：说明图捕获的难点**

   论文中提到的挑战可以用以下示例来说明。在早期的图捕获技术中，处理 Python 的原生逻辑（如控制流）和非 PyTorch 操作是非常困难的：

   Python

   ```
   import torch
   
   def example_logic(x):
       # 挑战 1: 数据依赖的控制流
       # 早期方案（如简单的 trace）可能只能记录下执行时的那条路径
       if x.sum() > 0:
           res = x * 2
       else:
           res = x / 2
       
       # 挑战 2: 与 Python 原生类型或外部库的交互
       # 转换为 Python 数值会使计算图在此时“断裂”
       threshold = res.mean().item() 
       
       # 挑战 3: 非 PyTorch 操作（如打印日志）
       # 静态图通常无法捕捉到这些副作用
       print(f"Current threshold: {threshold}")
       
       return res + threshold
   ```

### Lazy Tensors

(comparing with `torch.jit.trace`)

The Lazy Tensor approach and `torch.jit.trace` differ primarily in how they handle the relationship between the host language (Python) and the compiled graph.

According to the [Lazy Tensor paper](https://arxiv.org/pdf/2102.13267), here are the key differences:

#### 1. Implicit vs. Explicit Tracing

- **`torch.jit.trace`:** Requires an **explicit** call to trace a specific function with example inputs. It creates a static graph of that function which is then "lifted" out of Python.
- **Lazy Tensor:** Uses **implicit** tracing. It records operations behind the scenes during normal execution without requiring the user to wrap their code in specific tracing decorators or functions.

#### 2. Handling of Non-Tensor Code (The "Language Subset Problem")

- **`torch.jit.trace`:** Often suffers from the "language subset problem." If a traced function contains Python control flow (like `if` statements or `loops` that depend on data) or non-tensor operations (like using a third-party library), these are "baked in" based on the example inputs used during the trace. They will not update if the data changes in future runs.
- **Lazy Tensor:** Supports the **complete host language**. Because it maintains the "illusion" of eager execution, you can interleave arbitrary Python code, exceptions, and virtual function calls. If the code reaches a point where it needs to "see" the data (like a print statement or a data-dependent conditional), it simply triggers an evaluation of the current graph and then starts a new one.

#### 3. Execution Model

- **`torch.jit.trace`:** The goal is usually to create a serialized `ScriptModule` that can run independently of Python (e.g., in a C++ runtime).
- **Lazy Tensor:** Is a **high-performance backend** for the eager-mode API. It aims to give you the performance of a compiler (like XLA) while keeping you inside the flexible Python environment.

#### 4. Integration with Compilers

- **`torch.jit.trace`:** Primarily targets the TorchScript JIT compiler and optimizer.
- **Lazy Tensor:** Is designed specifically to target **Domain-Specific Compilers (DSCs)** like [XLA](https://www.tensorflow.org/xla). It groups sequences of operations into an IR graph that the DSC can optimize globally (e.g., through kernel fusion).

#### Summary Table

| **Feature**        | **torch.jit.trace**               | **Lazy Tensor**                       |
| ------------------ | --------------------------------- | ------------------------------------- |
| **Trigger**        | Explicit (user must call `trace`) | Implicit (automatic during execution) |
| **Control Flow**   | Captured/Fixed at trace time      | Handled by Python (Eager semantics)   |
| **Python Interop** | Limited within the traced block   | Full/Unrestricted                     |
| **Primary Goal**   | Model serialization/deployment    | Optimization via DSCs (like TPUs/XLA) |

In short, while `torch.jit.trace` tries to turn a piece of Python into a static graph, Lazy Tensor acts like a "smarter" eager mode that buffers commands into graphs only until it is forced to execute them.

#### How Lazy Tensors work with PyTorch2

In PyTorch 2.0, **LazyTensor** serves as a foundational "graph capture" technology that has been evolved to work alongside the new `torch.compile` stack. While `torch.compile` is the primary user-facing API for optimization, LazyTensor remains the core mechanism for targeting accelerators like TPUs via **PyTorch/XLA**.

##### 1. How it works in the PyTorch 2.0 Stack

In the context of the 2.0 release, LazyTensor functions in two distinct ways:

- **Standalone (Traditional XLA Mode):** You move tensors to a "lazy" device (`xm.xla_device()`). Operations are recorded into an Intermediate Representation (IR) graph. The graph is only compiled and executed when a "barrier" is hit (e.g., `mark_step()`, printing a tensor, or a data-dependent `if` statement).
- **As a `torch.compile` Backend:** This is the major 2.0 update. When you run `torch.compile(model, backend="openxla")`, PyTorch 2.0 uses **TorchDynamo** to intercept Python bytecode and extract an FX graph. It then hands this graph to the **LazyTensor** infrastructure to lower it into XLA HLO for compilation.

##### 2. Key Differences from Eager Mode

| **Feature**         | **Eager Mode (Standard)** | **LazyTensor (PT 2.0)**                             |
| ------------------- | ------------------------- | --------------------------------------------------- |
| **Execution**       | Immediate (Step-by-step)  | Deferred (Graph-based)                              |
| **Optimization**    | Local (Per-op)            | Global (Kernel fusion, memory planning)             |
| **Hardware Target** | Mostly CPU/GPU            | Primarily TPUs and specialized XLA hardware         |
| **Python Interop**  | Native                    | Triggers execution ("flushing") to sync with Python |

##### 3. Comparison with `torch.compile` (TorchDynamo)

The 2.0 "Dynamo" approach is generally preferred over "Pure Lazy" for the following reasons:

- **Reduced Tracing Overhead:** Standard LazyTensor re-traces and re-hashes graphs every iteration. The Dynamo-based `torch.compile` traces the model **once** at initialization and reuses the compiled binary, which is significantly faster for inference.
- **Handling Graph Breaks:** If your code contains unsupported Python (like a call to an external C library), Dynamo can "break" the graph and run that part in eager mode, whereas pure LazyTensor might require more manual intervention or cause performance "cliffs" when the graph is forced to execute prematurely.

##### 4. Why use LazyTensor in 2.0?

Despite the shift toward `torch.compile`, LazyTensor is still essential for **training on TPUs**. It allows for **async execution** where the CPU can stay ahead of the accelerator by recording the next step's graph while the current one is still running on the hardware.

For more technical details on the 2.0 integration, you can refer to the [PyTorch/XLA 2.0 release notes](https://pytorch.org/blog/pytorch-2-0-xla/) or the [TorchDynamo integration guide](https://docs.pytorch.org/xla/master/torch_compile.html).

### torch.fx.symbolic_trace

FX is a toolkit for developers to use to transform `nn.Module` instances. FX consists of three main components: a **symbolic tracer,** an **intermediate representation**, and **Python code generation**. A demonstration of these components in action:

```python
import torch


# Simple module for demonstration
class MyModule(torch.nn.Module):
    def __init__(self) -> None:
        super().__init__()
        self.param = torch.nn.Parameter(torch.rand(3, 4))
        self.linear = torch.nn.Linear(4, 5)

    def forward(self, x):
        return self.linear(x + self.param).clamp(min=0.0, max=1.0)


module = MyModule()

from torch.fx import symbolic_trace

# Symbolic tracing frontend - captures the semantics of the module
symbolic_traced: torch.fx.GraphModule = symbolic_trace(module)

# High-level intermediate representation (IR) - Graph representation
print(symbolic_traced.graph)
"""
graph():
    %x : [num_users=1] = placeholder[target=x]
    %param : [num_users=1] = get_attr[target=param]
    %add : [num_users=1] = call_function[target=operator.add](args = (%x, %param), kwargs = {})
    %linear : [num_users=1] = call_module[target=linear](args = (%add,), kwargs = {})
    %clamp : [num_users=1] = call_method[target=clamp](args = (%linear,), kwargs = {min: 0.0, max: 1.0})
    return clamp
"""

# Code generation - valid Python code
print(symbolic_traced.code)
"""
def forward(self, x):
    param = self.param
    add = x + param;  x = param = None
    linear = self.linear(add);  add = None
    clamp = linear.clamp(min = 0.0, max = 1.0);  linear = None
    return clamp
"""
```

`torch.fx` is a Python-only system for practical program capture and transformation in deep learning, designed to bridge the gap between imperative, eager-execution frameworks (like PyTorch) and the need for program structure capture for optimizations, analysis, and hardware integration. While early frameworks defined graph-based Intermediate Representations (IRs) for transformations, modern eager-mode frameworks prioritize productive development. `torch.fx` addresses this by providing a simplified, Python-centric approach focused on typical deep learning use cases rather than full language generality.

The core methodology of `torch.fx` is built upon three main components: program capture via symbolic tracing, a simple 6-instruction Python-based IR, and source-to-source Python code generation.

**Program Capture:**
`torch.fx` employs symbolic tracing, where operations on program values are recorded using a `Proxy` data structure. A `Proxy` is a duck-typed Python class that intercepts attribute accesses and method calls, acting as an abstract placeholder for concrete program values. This interception is notably achieved by leveraging PyTorch's `__torch_function__` protocol for operator dispatch and by overriding PyTorch's `Module` abstraction for module calls. Unlike example-based tracing, symbolic tracing does not specialize program flow based on concrete data-dependent control flow decisions, treating tensor metadata attributes like `shape` and `ndim` as `Proxy` values to be traced. This results in an Ahead-of-Time (AOT) capture process that is predictable and observable. The tracing mechanism is highly configurable via a `Tracer` class, allowing users to customize behavior by overriding methods like `is_leaf_module` (to specify opaque module calls) and `create_proxy` (to control node creation and associated runtime `Proxy` values, enabling custom metadata or data structures).

**Intermediate Representation (IR) Design:**
`torch.fx` represents captured programs as a Directed Acyclic Graph (DAG) using `Graph` objects, which contain a linear series of `Node` objects. This DAG-based IR is optimized for "basic block programs"—flat sequences of tensor operations common in deep learning models. The IR intentionally avoids primitive operations for data structure construction or mutation and does not include explicit control flow, which significantly simplifies analysis and transformation compared to richer IRs found in other frameworks.
Each `Node` has a string `opcode`, a `target` (e.g., the function/method/module to call), and `args` and `kwargs` (representing arguments in Python's calling convention as witnessed during tracing). Data dependencies between `Nodes` are represented by direct references to other `Nodes` within `args` and `kwargs`. Immediate Python built-in types (int, float, tuple, list) can appear directly as `Node` arguments.
The 6 fundamental `opcode` types are:
1.  `placeholder`: Represents a function input.
2.  `call_function`: Invokes a Python function specified by its `target` attribute.
3.  `call_method`: Invokes a method on `args[0]`, with the method name specified by `target`.
4.  `call_module`: Invokes the `forward` method of a sub-`torch.nn.Module`, identified by `target`.
5.  `get_attr`: Retrieves an attribute (typically a parameter or buffer) from the `Module` hierarchy, specified by `target`.
6.  `output`: Represents the return statement of the graph, with `args[0]` being the return value.
The `Graph` and module parameters are encapsulated within a `GraphModule`, which is a subclass of `torch.nn.Module`. This design allows transformed code to interact with the PyTorch ecosystem seamlessly, enabling natural manipulation of mutable state (like model parameters) alongside code transformations.

**Source-to-Source Transformation:**
The final stage is code generation. `torch.fx` generates valid Python source code directly from the transformed IR. This generated code is then loaded into Python, producing a callable Python object that replaces the `forward` method of the `GraphModule` instance. This approach ensures that the results of `torch.fx` transformations remain within the Python ecosystem, making them inspectable, debuggable (e.g., with `pdb`), compatible with other Python libraries, and amenable to further transformations.

**Case Studies and Evaluation:**
`torch.fx`'s design choices have enabled various applications:
*   **IR Complexity Reduction**: For a canonical ResNet50, `torch.fx` produces an IR with 445 operations, significantly less verbose than `torch.jit.script` (2614 operations) and `torch.jit.trace` (860 operations), due to its focus on essential tensor operations and inlining constants.
*   **Performance Optimization**:
    *   **Quantization**: `torch.fx` facilitates Post-Training Quantization and Quantization-Aware Training by providing the necessary program representation for instrumenting "observer" objects and converting operations. This led to up to a 3.3x runtime performance improvement for a DeepRecommender model on Intel Xeon Gold CPUs using FBGEMM.
    *   **Operator Fusion**: Enables transformations like Convolution-BatchNorm fusion by providing non-local program context and state modification. For a PyTorch ResNet50, this yielded approximately a 6% latency reduction on GPU and up to 40% on CPU.
    *   **Program Scheduling**: Allowed reordering of blocking remote procedure calls in large distributed training jobs, increasing QPS by up to 9%.
*   **Program Analysis**: Utilized for simulating deep learning inference to estimate FLOPs, memory bandwidth usage, and data value sizes. It also provides a foundation for shape analysis (`fx.passes.shape_prop`) and graph visualization (`fx.graph_drawer`).
*   **Device and Runtime Export/Compilation**: Serves as a pathway for integrating PyTorch models with specialized compilers and hardware backends like NVIDIA TensorRT. An experimental `torch.fx`-to-TensorRT lowering system demonstrated 3.7x runtime speed-up for ResNet50 and 1.54x for LearningToPaint models on NVIDIA Tesla V100 GPUs.

In summary, `torch.fx` prioritizes ease of use and developer productivity for ML practitioners by offering a simplified, Python-native framework for capturing and transforming deep learning programs, effectively enabling various performance optimizations, program analyses, and hardware integrations.

## 三。PyTorch 2 的核心技术

为了解决上述矛盾，本文介绍了 PyTorch 2 的核心功能 `torch.compile` 背后的两个扩展：

- **TorchDynamo：** 一个 Python 级别的 JIT（即时）编译器。它通过挂钩 CPython 的帧评估 API，在执行前动态修改字节码，从而在保留 Python 灵活性的同时，提取出 PyTorch 操作序列并生成 FX 图。
- **TorchInductor：** TorchDynamo 的默认编译器后端。它将 PyTorch 程序翻译成 GPU 上的 OpenAI Triton 代码以及 CPU 上的 C++/OpenMP 代码。它使用 Python 编写，易于扩展。

### `__torch_function__`

The **`__torch_function__`** protocol is a mechanism in PyTorch that allows "duck-typed" objects (objects that act like tensors but aren't actually `torch.Tensor` instances) to define how they should behave when passed into standard PyTorch functions like `torch.relu()`, `torch.add()`, or `torch.matmul()`.

As mentioned in the [torch.fx paper](https://arxiv.org/pdf/2112.08429), this protocol is a cornerstone of how **symbolic tracing** works.

#### How it Works

When you call a PyTorch function—for example, `torch.sigmoid(x)`—PyTorch checks the arguments. If any argument (like `x`) defines a `__torch_function__` method, PyTorch hands control over to that method instead of running the standard C++ implementation.

The method signature typically looks like this:

Python

```
def __torch_function__(cls, func, types, args=(), kwargs=None):
    # 'func' is the torch function being called (e.g., torch.sigmoid)
    # 'args' and 'kwargs' are the inputs passed to it
    ...
```

#### Why it Matters for `torch.fx`

The `torch.fx` library uses this protocol to "intercept" calls. When you trace a model:

1. `torch.fx` creates **Proxy** objects. These are not real tensors; they are placeholders.
2. When your model code does `torch.relu(proxy_hidden_state)`, the **`__torch_function__`** inside the Proxy is triggered.
3. Instead of doing math, the Proxy records: *"Someone just called 'relu' on me."*
4. It adds a new "node" to the [torch.fx Graph](https://arxiv.org/pdf/2112.08429) and returns a *new* Proxy representing the result of that operation.

#### Use Cases Beyond `torch.fx`

The protocol is widely used in the PyTorch ecosystem for:

- **Metadata Propagation:** Carrying extra information (like physical units or named dimensions) through a series of operations.
- **Distributed Computing:** Wrapping tensors to automatically handle data moving between different machines.
- **Logging/Debugging:** Creating "Tensors" that log every mathematical operation performed on them for research purposes.

By using this protocol, [torch.fx](https://arxiv.org/pdf/2112.08429) can capture the structure of a model without requiring the user to rewrite their code or use a special domain-specific language.

### symbolic shapes reasoning

In the context of the [PyTorch 2 paper](https://dl.acm.org/doi/epdf/10.1145/3620665.3640366), **symbolic shapes reasoning** refers to the compiler's ability to represent tensor dimensions as mathematical variables (e.g., $s_0$, $s_1$) rather than fixed integers.

This allows the compiler to handle **dynamic shapes**—situations where input sizes (like batch size or sequence length) change between runs—without requiring a slow recompilation every time a dimension fluctuates.

Key aspects of this system include:

#### 1. Symbolic Shape Guards

Instead of capturing a graph that only works for a specific size (e.g., $64 \times 128$), TorchDynamo captures a trace where some dimensions are symbolic. It maintains **guards** (logical assumptions) about these symbols. If a new input arrives that satisfies the existing guards, the compiled kernel can be reused safely.

#### 2. Meta Functions

To reason about shapes without actually performing heavy tensor computations, PyTorch uses **meta functions**. These are small snippets of code for every operator that calculate the *output shape* based on *input shapes*. For example, if you concatenate two tensors of size $s_0$ and $s_1$ along dimension 0, the meta function tells the compiler the result is $s_0 + s_1$.

#### 3. Optimization and Simplification

Reasoning over complex algebraic expressions can become slow. To mitigate this, the system:

- **0/1 Specialization:** It treats dimensions that are 0 or 1 as constants rather than symbols to simplify broadcasting logic.
- **SymPy Integration:** It uses the [SymPy](https://dl.acm.org/doi/epdf/10.1145/3620665.3640366) library to algebraically simplify shape expressions and determine if a new guard is already logically covered by previous ones.

#### 4. Unbacked Symbolic Integers

Sometimes a shape depends on the *data* inside a tensor (e.g., the output of `.nonzero()`). These are called "unbacked" symbols because their value isn't known until the code actually runs. The reasoning system must track these carefully to avoid illegal control flow (branching based on an unknown value) while still allowing the compiler to generate efficient code for the rest of the graph.

### meta functions

Meta functions exist to answer: **"Given these input shapes/dtypes, what is the output shape/dtype?"** — without doing any actual computation or allocating any data.

They're used in three main contexts:

------

#### 1. `torch.compile` / Dynamo + Inductor

When tracing a model, Dynamo needs to know the shape and dtype of every intermediate tensor to generate code. It uses **FakeTensor** — a tensor subclass with no storage — and runs meta functions to propagate shapes through the graph symbolically. Without a meta function, the compiler can't trace through the op.

#### 2. `torch.export`

Export traces the full graph ahead-of-time. Same requirement: every op must have a meta function so shapes can be inferred statically (or symbolically with `SymInt`).

#### 3. Meta device tensors

Users can explicitly create tensors on `device="meta"` to reason about model memory/parameter counts without allocating GPU memory. Meta functions make ops runnable on such tensors.



```python
x = torch.empty(1024, 1024, device="meta")
y = torch.mm(x, x)  # works — meta function computes output shape
print(y.shape)       # torch.Size([1024, 1024])
```

------

#### What they are NOT

- Not a decomposition — they don't define *how* the op computes, only *what shape/dtype comes out*
- Not for correctness testing — the output tensor has no data
- Not needed for eager execution on real devices (CPU/CUDA have their own kernels)

------

#### TL;DR

Meta functions are the **shape inference system** for the compiler and symbolic execution stack. Any op that needs to participate in `torch.compile`, `torch.export`, or `device="meta"` must have one.

------

### Real-World PyTorch Model Compatibility CI Systems

#### 1. **TorchBench** (Meta / PyTorch team) — `pytorch/benchmark`

The official PyTorch benchmark and CI suite.

- A collection of open-source benchmarks used to evaluate PyTorch performance. `torchbenchmark/models` contains copies of popular or exemplary workloads modified to expose a standardized API for benchmark drivers and optionally enable backends such as TorchInductor/TorchScript.

- TorchBench consists of 84 models from various domains. Based on GitHub Actions, they create a series of GitHub workflows to continuously test performance regression with TorchBench — this was the first such CI effort for the PyTorch repository.

- Currently, models run on nightly PyTorch builds and push data to Meta's internal database. The Nightly CI publishes both V1 and V0 performance scores.

- Integrating it into your own CI is straightforward:

  ```python
  from torchbenchmark.models import stable_diffusion_text_encoderbenchmark = Model(test="eval", device="cuda")model, example = benchmark.get_module()model(*example)
  ```

------

#### 2. **torch.compile Compatibility Suite** (PyTorch 2.0 launch)

Used to validate `torch.compile` across a massive model corpus before release.

- PyTorch ran `torch.compile` on 163 open-source models from HuggingFace, TIMM, and TorchBench to evaluate performance gains across tasks like Image Classification, Object Detection, Image Generation, NLP (Language Modeling, Q&A, Sequence Classification), Recommender Systems, and Reinforcement Learning. `torch.compile` worked on 93% of the models, achieving 43% faster execution on NVIDIA A100.

------

#### 3. **Hugging Face Transformers CI** — model zoo testing

- Any changes to the modeling or PyTorch examples code requires running the model zoo tests. Tests requiring GPUs run on a different CI nightly (since PR CI doesn't have GPUs). Slow tests run on a scheduled basis rather than in PR CI checks.
- They use a `@slow` decorator pattern to skip heavy model tests in PRs and run them nightly instead — a common pattern for large model suites.

------

#### 4. **BackendBench** (GPUMODE)

- Provides operators and inputs derived from 155 model traces found in TIMM (67), HuggingFace Transformers (45), and TorchBench (43). These are also the models PyTorch developers use to validate performance.

------

#### 5. **Common Infrastructure Push** (torchbench + TIMM + HuggingFace)

- There has been active work on a common benchmarking infrastructure that allows getting comparable results for different backend technologies. TorchBench already installs HF and TIMM repos as part of its setup, but models and configs not part of TorchBench are not easy to access.
- Running each model in a subprocess is recommended — it adds ~10s per model initialization but is a worthy price for reliable, isolated results. The `--isolate` flag in torchbench.py enables subprocess isolation.

------

#### Key Design Patterns from Production Systems

| Pattern                       | How it's done                                                |
| ----------------------------- | ------------------------------------------------------------ |
| **Tiered testing**            | Fast tests on every PR; slow/GPU tests nightly               |
| **Subprocess isolation**      | Each model runs in its own process to prevent state contamination |
| **Standardized model API**    | All models expose `get_module()` → `(model, example_inputs)` |
| **Multi-source model corpus** | Pull from TorchBench + TIMM + HuggingFace for broad coverage |
| **Nightly vs. PR gates**      | Only ~10–20 representative models block PRs; full suite runs overnight |

The PyTorch team's approach of testing across 163+ models before a major release (2.0) is essentially the gold standard for this kind of compatibility CI at scale.

## 五。torch_npu

### What it is

This is **not** a fork of PyTorch itself. It is the **Ascend Extension for PyTorch** (`torch_npu`), a
 separate Python package developed by Huawei's Ascend team. It installs on top of standard PyTorch an
d adds support for **Huawei Ascend NPU** hardware (Atlas 800, 900, etc.).

The current version is `2.11.0`, tracking PyTorch releases back to 1.5.0 and up through 2.8+.

---

### Architecture

Instead of patching PyTorch core, `torch_npu` hooks into PyTorch's **backend extension APIs** and reg
isters the NPU as a first-class device. Key subsystems:

| Subsystem                                                    | Location                                    | What it does                                                 |
| ------------------------------------------------------------ | ------------------------------------------- | ------------------------------------------------------------ |
| **ATen ops**                                                 | `torch_npu/csrc/aten/`                      | NPU kernel registrations for standard PyTorch ops (binary ops, autocast, pin memory, etc.)                              |                                             |                                                              |
| **Distributed**                                              | `torch_npu/csrc/distributed/`               | HCCL (Huawei Collective Communications Library) and LCCL process groups, replacing NCCL for multi-device/multi-node training |                                             |                                                              |
| **NPU runtime**                                              | `torch_npu/npu/`                            | Device management, memory, streams, graphs, random state, AMP/autocast                                                     |                                             |                                                              |
| **Dynamo/Inductor**                                          | `torch_npu/dynamo/`, `torch_npu/_inductor/` | `torch.compile()` backend support for NPU                                                   |                                             |                                                              |
| **Compiler (TorChair)**                                      | `third_party/torchair/`                     | Graph-mode compiler bridge — converts traced PyTorch graphs to Ascend GE (Graph Engine) format              |                                             |                                                              |
| **Op Plugin**                                                | `third_party/op-plugin/`                    | Ascend-specific operator implementations                     |
| **Profiler**                                                 | `torch_npu/profiler/`                       | NPU profiling tools integrated with PyTorch's profiler interface |

---

### Key differences from mainline PyTorch

1. **Device backend**: Introduces `npu` as a device (analogous to `cuda`). Tensors can be moved to NP
U with `.npu()` instead of `.cuda()`.

2. **Collective comms**: Replaces NCCL with **HCCL** (`ProcessGroupHCCL`) and **LCCL** (`ProcessGroup
LCCL`) for distributed training. Has its own rendezvous, TCP store, and parallel store proxy implemen
tations.

3. **Compiler stack**: Integrates **TorChair** (graph compiler) and **torchmlir** as third-party subm
odules, enabling `torch.compile()` to target Ascend's GE graph engine rather than Triton/CUDA.

4. **Custom data types**: Adds `HiFloat8Tensor` and custom dtype support (`torch_npu/csrc/custom_dtyp
e/`).

5. **CANN dependency**: Requires Huawei's **CANN** (Compute Architecture for Neural Networks) toolkit
 — the NPU runtime libraries (`libascendcl.so`, `libhccl.so`, etc.).

6. **Autoloading**: Uses `TORCH_DEVICE_BACKEND_AUTOLOAD` so that importing `torch` can automatically load `torch_npu` from PyTorch 2.5.1+, without an explicit `import torch_npu`.

7. **Format management**: Has NPU-specific tensor format/layout support (`torch_npu/npu/_format.py`), relevant for Ascend's internal memory layouts.

---

### Relationship to upstream PyTorch

- Tracks official PyTorch releases (does not diverge the core)
- Maintained in sync via CI that runs daily against `pytorch/pytorch:main`
- Versioned as `{pytorch_version}-{ascend_version}` (e.g., `v2.6.0-7.3.0`)
- CANN version must be matched to the extension version
